DEEP DIVE: The Double Booking Problem

Problem: multiple users trying to book a particular seat

Optimistic Locking:
Works for low/moderate contention, but BookMyShow hotspot traffic has extreme contention.
10,000 users hitting same seat.
all read available seat, all attempt update
1 succeeds, 9999 fail

Pessimistic Locking:
User selects seat 1,2,3 (should be available)
store using setnx in redis: 
    SET show:999:seat:1 booking123 NX EX 600
    SET show:999:seat:2 booking123 NX EX 600
    SET show:999:seat:3 booking123 NX EX 600

If any lock fails: rollback acquired locks

Persist in DB:

Seats:
1 -> LOCKED
2 -> LOCKED
3 -> LOCKED

Booking:
booking123
status = PENDING

Payment Succeeds:
    Update DB:
    seat1: Booked
    seat2: Booked
    seat3: Booked

Payment Fails:
    Invalidate Cache
    Update DB:
    seat1: Available
    seat2: Available
    seat3: Available

Scaling:

Services are stateless and can be horizontally scaled independently behind a load balancer.
For relatively static search metadata (movies, theatres, shows), add a Redis cache since this data changes infrequently.

Seat Map Scaling
Seat map traffic can be very high, especially for popular shows.
A naive approach of hitting the primary seats DB for every request would overload it.

Approach:
Add read replicas for scaling seat availability reads
But seat map freshness matters, since seats may be getting locked in real time

Seat availability = Seats DB state (price, data, etc) + active Redis locks (for availibility)
